사용자가 서비스를 사용하면서 느끼는 전체적인 경험. 단순히 '예쁜 화면'이 아니라 쉽게 찾을 수 있는지·불편함이 없는지·만족감이 있는지 등 모든 경험을 포함합니다.
📌 예: 로그인 과정이 너무 복잡하다 → UX가 나쁘다
UI
User Interface · 사용자 인터페이스
▼
사용자가 직접 보고 누르는 화면의 요소. 버튼·메뉴·색상·폰트·레이아웃 등 시각적·조작적 요소 전체를 말합니다. UX가 '경험 전체'라면 UI는 그 경험을 만드는 '화면 요소'입니다.
📌 예: 파란색 버튼, 상단 메뉴바, 카드 레이아웃
IA
Information Architecture · 정보 구조
▼
서비스 안의 모든 콘텐츠를 어떻게 분류하고 배치할지 설계하는 것. 도서관의 서가 배치처럼 사용자가 원하는 정보를 쉽게 찾을 수 있도록 구조를 잡습니다.
📌 예: 홈 > 탐색 > 카테고리 > 상세 페이지 구조
Global Navigation
Global Navigation
▼
모든 페이지 상단에 공통으로 있는 메뉴바. '로고 + 주메뉴 + 검색 + 로그인' 구성이 일반적입니다. 어느 페이지에 있어도 항상 보이는 공통 탐색 도구입니다.
📌 예: 네이버 상단의 서비스 메뉴 바
Local Navigation
Local Navigation
▼
특정 섹션 안에서만 보이는 서브 메뉴. 카테고리 탭이나 사이드바 형태로 나타납니다. Global Navigation이 전체 사이트의 지도라면 Local Navigation은 현재 위치의 세부 지도입니다.
📌 예: 쇼핑몰 카테고리 탭: 상의 | 하의 | 아우터
CTA
Call to Action · 행동 유도 버튼
▼
사용자가 특정 행동을 하도록 유도하는 버튼이나 링크. '지금 시작하기', '무료 체험', '구매하기' 같은 문구가 CTA입니다. 전환율에 가장 직접적 영향을 줍니다.
📌 예: 랜딩페이지의 '지금 무료로 시작하기' 버튼
KSF
Key Success Factor · 핵심 성공 요인
▼
경쟁사 분석 후 '우리 서비스가 성공하기 위해 반드시 갖춰야 할 요소'를 도출한 것. 경쟁사가 못하는 것을 우리가 잘하면 그게 KSF가 됩니다.
📌 예: 맛집 앱에서 '또래 추천 기반 신뢰도'가 KSF
MVP
Minimum Viable Product · 최소 기능 제품
▼
핵심 기능만 담아 가장 빠르게 출시하는 첫 번째 버전. 완벽한 완성품을 만들기 전에 최소한의 기능으로 사용자 반응을 먼저 확인하는 전략입니다.
📌 예: 배달앱 MVP: 주문·결제만 먼저, 리뷰는 나중에
SEO
Search Engine Optimization · 검색엔진 최적화
▼
구글·네이버 같은 검색엔진에서 내 사이트가 상위에 노출되도록 최적화하는 작업. 타이틀 태그, 메타 설명, 헤딩 구조, 키워드 배치가 핵심입니다.
📌 예: 타이틀 태그에 '서울 맛집 추천' 키워드 포함
와이어프레임
Wireframe · 화면 구조 스케치
▼
실제 디자인 전에 화면의 레이아웃과 구성 요소를 단순한 박스와 선으로 그린 설계도. 색상·이미지 없이 어디에 무엇이 들어갈지 구조만 표현합니다.
📌 예: 박스로만 그린 앱 메인 화면 레이아웃
페르소나
Persona · 가상 사용자 캐릭터
▼
사용자 인터뷰 결과를 바탕으로 만든 가상의 대표 사용자. 이름·나이·직업·불편함·목표를 구체적으로 설정해 '누구를 위해 만드는가'를 명확히 합니다.
📌 예: 김지수(22세, 대학생, 신뢰할 수 있는 맛집을 빠르게 찾고 싶음)
스토리보드

▼

🧭 바이브코딩 핵심 원칙 6가지 — 처음 시작 전 반드시 읽기
① 전체 방향을 먼저 설명하기
세부 기능보다 무엇을·왜·핵심 조건을 먼저 전달하세요. AI가 구조와 방향을 제안한 뒤 세부 기능을 붙여나가는 게 안정적입니다.
② 구현 전 계획 먼저 확인하기
바로 코드 요청 전에 "어떻게 구현할지, 어떤 순서로, 예상 위험은?"을 먼저 물어보세요. 시행착오를 크게 줄입니다.
③ 작은 단위로 나눠서 진행하기
로그인 → 회원가입 → 결제처럼 한 번에 하나씩 완성하고 확인한 뒤 다음 단계로 넘어가세요. 한 번에 많이 시키면 일부가 빠집니다.
④ AI 작업 범위를 미리 정하기
자유롭게 해도 되는 작업(코드 정리·문서), 확인 필요(DB 구조 변경), 절대 건드리면 안 되는 것(API Key·민감 정보)을 사전에 명시하세요.
⑤ 사람이 항상 최종 검토하기
AI 결과는 완성본이 아닌 초안입니다. 겉은 멀쩡해 보여도 오류가 숨어있을 수 있습니다. 기능 동작·요구사항 누락·비정상 흐름은 반드시 직접 확인하세요.
⑥ 처음부터 완벽하게 X → 빠르게 만들고 개선
바이브코딩의 강점은 빠르게 만들고 계속 개선하는 것입니다. 먼저 간단하게 작동하게 만들고, 쓰면서 수정해 나가세요.

3. 디지털 파일 제출 — JPG + PSD
인쇄소 납품이 아닌 파일 제출이므로 색상 모드·해상도 설정이 다릅니다

      

    


    

      📁 제출 형식  
      PSD
      +
      JPG
      PSD는 레이어 유지 원본, JPG는 최종 결과물 이미지로 각각 저장해서 함께 제출

      🎨 색상 모드 — 디지털 제출이므로 RGB 모드로 작업해도 됩니다. 화면에서 보이는 색상이 제출 결과물과 동일합니다.

      CMYK는 인쇄소 납품용 설정 — 나중에 실제 인쇄할 계획이 있을 때 고려하면 됩니다.
    


    

      

        
01

        

          
📐 해상도 — 72 DPI vs 150 vs 300

          
    
            
  • ▸ JPG 제출용: 150~200 DPI 권장 — 화면에서 선명하고 파일 용량 적당
    • 
            
  • ▸ 나중에 실제 인쇄까지 고려한다면 300 DPI로 설정 (파일 용량 커짐)
    • 
            
  • ▸ 72 DPI는 웹 전용 — JPG 화질이 떨어져 보임, 비추천
    • 
            
  • ▸ 캔버스 크기는 달력 종류에 맞게 설정 (아래 사이즈 가이드 참고)
    • 
          

          

            권장: RGB / 150~300 DPI — 제출용이면 150 DPI도 충분
          

        

      


      

        
02

        

          
🔤 폰트 — 다른 PC에서 열 때 깨짐 주의

          
    
            
  • ▸ PSD를 다른 컴퓨터에서 열면 폰트가 없어 깨질 수 있습니다
    • 
            
  • ▸ PSD 저장 전 텍스트 레이어 래스터화 권장 (우클릭 → 레이어 래스터화)
    • 
            
  • ▸ 래스터화 후엔 글자 수정 불가 — 내용 확정 후 마지막에 처리
    • 
            
  • ▸ 저작권 있는 폰트는 무료 상업용 라이선스 여부 반드시 확인
    • 
          

          

            순서: 내용 확정 → 텍스트 래스터화 → PSD 저장 → JPG 내보내기
          

        

      


      

        
03

        

          
📋 날짜 · 요일 · 공휴일 최종 검수

          
    
            
  • ▸ 날짜 오류는 제출 후 수정 불가 — 가장 치명적인 실수
    • 
            
  • ▸ 선택한 달의 공휴일을 인터넷에서 반드시 검색·확인 후 입력
    • 
            
  • ▸ 요일 시작(일요일/월요일)을 2개월 모두 동일하게 통일
    • 
            
  • ▸ 타인에게 교차 검증 요청 (눈은 익숙한 것을 놓침)
    • 
          

          

            제출 전 체크: 날짜 → 요일 → 공휴일 → 폰트 깨짐 → JPG 화질

2. 기획 의도 작성 방법 — HOW
디자인 시작 전에 반드시 이 순서로 생각을 정리하세요

      

    


    
    

      

        

          
1

          목적 정하기
        

        

          
이 달력은 무엇을 위해 만드는가?

          

            예: 가족에게 선물할 개인 소장용 달력

            예: 카페 홍보용 상업 달력

            예: 환경 캠페인 공익 달력
          

        

      

      

        

          
2

          타깃 정하기
        

        

          
누가 이 달력을 보는가?

          

            연령·성별·취향·사용 환경을 구체적으로

            예: 30~40대 워킹맘, 주방에 붙여두는 용도

            예: 20대 대학생, 책상 위 탁상달력
          

        

      

      

        

          
3

          콘셉트 키워드
        

        

          
느낌을 3개 단어로 표현하기

          

            예: 따뜻한 · 포근한 · 가족적인

            예: 세련된 · 미니멀한 · 현대적인

            예: 자연스러운 · 풋풋한 · 싱그러운
          

        

      

      

        

          
4

          색상·폰트 방향
        

        

          
키워드에서 시각 요소 도출하기

          

            따뜻한 → 주황·베이지 계열

            세련된 → 흑백·남색 + 산세리프 폰트

            자연스러운 → 초록·갈색 + 손글씨 폰트

픽셀 (Pixel)

    • 
        
  • PPI (Pixels Per Inch): 1인치 안의 픽셀 수 → 화면 선명도
    • 
        
  • DPI (Dots Per Inch): 프린터 출력 해상도 단위
    • 
        
  • 웹: 72~96 PPI / 인쇄: 300 DPI 이상 권장
    • 
        
  • 포토샵: 새 파일 시 해상도 반드시 확인!

⚡ 바이브코딩 (Vibe Coding) — AI로 웹·앱 구현하기

    
"바이브코딩"이란 코딩을 몰라도 AI에게 대화하듯 말해서 웹·앱을 만드는 새로운 개발 방식입니다.

    2024~2025년 실리콘밸리에서 폭발적으로 확산된 트렌드로, 디자이너·기획자도 프로토타입을 직접 구현할 수 있게 됩니다.

    웹기획과디자인 수업의 결과물을 실제 작동하는 프로토타입으로 발전시킬 수 있는 핵심 스킬입니다.

  


  

    
🔄

2
💬 프롬프트 작성

3
🤖 AI 코드 생성

4
🔁 반복 수정
"버튼 색을 파란색으로 바꿔줘", "모바일에서도 잘 보이게 해줘" 처럼 대화하듯 수정을 반복합니다.

    
5
🌐 배포·공유
Vercel, Netlify, GitHub Pages 등 무료 호스팅 서비스로 1분 안에 URL을 발급받아 팀원·교수에게 공유합니다.

# 바이브코딩 프롬프트 예시

부산 맛집 추천 앱의 메인 화면을 HTML/CSS로 만들어줘.

- 상단에 검색바 (돋보기 아이콘 포함)
- 카테고리 버튼: 한식, 중식, 일식, 카페, 술집 (가로 스크롤)
- 맛집 카드 3개 (이미지 영역 + 가게명 + 별점 + 거리)
- 하단 내비게이션 바: 홈, 검색, 저장, 마이

디자인:
- 메인 컬러: #FF6B35 (따뜻한 오렌지)
- 폰트: Noto Sans KR
- 모바일 우선 (390px 기준)
- 카드에 그림자 효과
